\section{BiTempQA Benchmark}

\subsection{Problem Formulation}
Following bitemporal data modeling~\citep{snodgrass1999tsql2}, we define two temporal dimensions:
\begin{itemize}[leftmargin=*]
\item \textbf{Event Time} ($t_e$): When an event actually occurred in the real world.
\item \textbf{Record Time} ($t_r$): When the memory system recorded (learned about) the event.
\end{itemize}
A memory entry is a tuple $m = (\text{text}, t_e, t_r)$, and a memory store at logical time $T$ contains all entries with $t_r \leq T$. A question may require reasoning about $t_e$ only, $t_r$ only, or both simultaneously.

\subsection{Scenario Design}
We design \textbf{10 scenario types} spanning 6 domains (corporate, academic, social, fictional, historical, scientific):
\begin{enumerate}[leftmargin=*]
\item \textbf{Entity Attribute Evolution}: Attributes change over time (e.g., job titles).
\item \textbf{Relationship Evolution}: Social/professional relationships change.
\item \textbf{Contradictory Information}: Different sources report conflicting facts.
\item \textbf{Late-Arriving Facts}: Events recorded well after they occurred ($|t_r - t_e|$ large).
\item \textbf{Future-Dated Information}: Events recorded before their stated occurrence ($t_r < t_e$).
\item \textbf{Entity Identity Resolution}: Same entity referenced differently across time.
\item \textbf{Knowledge Retraction}: Previously recorded facts are later revoked.
\item \textbf{Multi-Source Information}: Multiple independent sources report overlapping events.
\item \textbf{Gradual Accumulation}: Knowledge builds incrementally over time.
\item \textbf{Temporal Ambiguity}: Unclear or ambiguous temporal references.
\end{enumerate}

Each scenario consists of 3--5 memory writes, each annotated with explicit $t_e$ and $t_r$ ISO~8601 timestamps. Scenarios are designed so that $t_e \neq t_r$ in the majority of cases, creating natural bitemporal tension.

\subsection{Question Taxonomy}
Questions are organized into \textbf{9 types} at \textbf{3 difficulty levels}:

\textbf{Level 1 (Easy)}: Point-in-time retrieval, temporal ordering, first-recorded detection.

\textbf{Level 2 (Medium)}: Period queries, change detection, multi-hop temporal reasoning.

\textbf{Level 3 (Hard)}: Counterfactual reasoning, version conflict resolution, complex temporal composition.

Questions use three answer formats: multiple choice (4 options, 43.8\%), abstractive (free text, 53.0\%), and boolean (3.2\%). Every question is annotated with metadata flags: \texttt{requires\_event\_time}, \texttt{requires\_record\_time}, \texttt{requires\_version\_tracking}, and \texttt{requires\_knowledge\_retraction}.

All QA pairs are generated via Qwen2.5-72B-Instruct with structured prompts, then validated for parseability and temporal consistency. The dataset is Chinese-language, reflecting the growing importance of multilingual NLP evaluation.

\subsection{Evaluation Pipeline}
We implement a unified \textbf{Retrieve$\to$Generate$\to$Judge} pipeline (Figure~\ref{fig:framework}):

\textbf{Step 1---Retrieve}: The memory backend retrieves the top-$k$ context passages given the question and scenario memory store.

\textbf{Step 2---Generate}: DeepSeek-V3~\citep{deepseekv3} generates an answer conditioned on the retrieved context.

\textbf{Step 3---Judge}: An independent LLM call scores the generated answer against the gold answer. For MC questions, exact letter match is used (100\% parse rate). For abstractive and boolean questions, DeepSeek-V3 in \texttt{answer\_judge} mode evaluates semantic correctness with lenient matching. Without the LLM judge, abstractive accuracy drops to ${\sim}10\%$ (text matching too strict); with it, accuracy is ${\sim}75\%$ (fair semantic evaluation).
